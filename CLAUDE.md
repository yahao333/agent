# Ralph — Working Agreement for Claude Code

> Read this first whenever you work on Ralph. It captures the contract between
> the human (Yang) and Claude Code so we don't relitigate decisions every session.

## 1. What is Ralph

Ralph is a Go CLI that runs an **autonomous task loop on top of Claude Code**.
Given a goal, Ralph repeatedly invokes `claude -p` until the LLM self-reports
done AND an external verifier (e.g. `make test`) passes — or a safety limit
triggers.

Ralph is the **orchestrator**, not the LLM. All actual coding is done by
Claude Code under Ralph's supervision.

## 3. Architecture in one paragraph

`cmd/ralph` → `internal/agent.Loop` (state machine) → `internal/executor.Executor`
(invokes `claude` CLI, parses stream-json) → `internal/memory` (scratchpad +
state.json) → `internal/verify` (runs `verify_cmd`). State is persisted under
`.ralph/runs/<run_id>/` after **every** state transition. session_id is
generated by Ralph (UUID) and passed via `--session-id` (first turn) /
`--resume` (later turns).

## 4. Hard rules

### 4.1 The Claude Code interface is sacred

We invoke Claude Code with a **fixed** flag set (see
`claude-code-integration.md` §1.3). Do not add, remove, or change these flags
without updating the doc and an ADR.

The stream-json parser must tolerate **unknown event types** — log and skip,
never crash. New event types from future Claude Code versions must not break us.

### 4.2 State block extraction is forgiving

Per `scratchpad-protocol.md` §4.4: missing/malformed state blocks **must not**
abort the Run. Fall back to `in_progress` and log the error to
`state.json.last_extract_error`.

### 4.3 Persist before transition

Every FSM state transition must call `StateStore.Update()` **before** doing
the work of the new state. This makes kill -9 recovery possible (v2 feature).

### 4.4 Cost limits are enforced by Claude Code

We pass `--max-budget-usd` and trust Claude Code to enforce it. Don't try to
count tokens ourselves.

### 4.5 The verifier is external, not Ralph's problem

Ralph runs `verify_cmd` (user-configured or B2 fallback to `make test`).
Ralph does NOT know what "tests passing" means. Exit code 0 = pass, anything
else = fail. Output is captured verbatim and fed back to the LLM.

## 5. How to add a feature

1. If it touches the state machine: update `agent-loop.md` first
2. If it touches Claude Code flags: update `claude-code-integration.md` first
3. If it changes scratchpad format: update `scratchpad-protocol.md` first
4. If it's a non-trivial design decision: write an ADR
5. Code last; tests alongside; `make lint test` before commit
6. Conventional Commits: `feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:`

## 6. How to add a Claude Code feature dependency

We currently depend on these CLI flags (per integration spec):
`--session-id`, `--resume`, `--output-format stream-json`, `--verbose`,
`--permission-mode`, `--max-budget-usd`, `--append-system-prompt`, `-p`.

If you want to use another flag (e.g. `--json-schema`, `--fork-session`,
`--include-partial-messages`):

1. Verify it actually exists in the current `claude --help`
2. Run a smoke test, capture the output, paste into the PR
3. Add a row to `claude-code-integration.md` §1.4 (optional flags)

## 7. Things Ralph deliberately does NOT do (v1)

- Multi-agent / concurrent Runs
- Resume an interrupted Run (`ralph resume <id>`)
- Direct API calls to Anthropic/OpenAI
- Token counting (delegated to Claude Code's budget)
- Prompt rewriting / self-reflection
- Sandboxing (use git + `bypassPermissions` warnings)

If a feature request falls in this list, push back and reference this section.

## 8. Testing strategy

- **Unit tests**: state machine transitions, state-block parser (lots of edge
  cases here), config loader, verify fallback ladder
- **Integration test (slow, opt-in)**: runs `claude -p` for real on a toy goal.
  Tagged `//go:build integration` so `make test` skips it by default.
  Run with `go test -tags=integration ./...`
- **Mock executor**: `internal/executor/mock.go` (to be added) for fast loop tests

## 9. Logs and observability

- Per iteration: `.ralph/runs/<id>/iterations/NNN/events.jsonl` (raw stream),
  `stderr.log`, `thinking.log`, `tools.log`
- Per verify: `.ralph/runs/<id>/verify/NNN.log`
- Global state: `.ralph/runs/<id>/state.json`
- Final outcome: `.ralph/runs/<id>/result.json`

When debugging a Run, **start from `state.json`**, then drill into the
iteration that misbehaved.

## 10. When in doubt

Ask. Better to spend 5 minutes clarifying than to spend 5 hours implementing
the wrong thing. There's no penalty for "I don't know" — the penalty is for
silent assumptions.
